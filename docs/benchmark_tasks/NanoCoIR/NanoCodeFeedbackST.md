## Overview

`NanoCodeFeedbackST` evaluates single-turn code-assistant retrieval. Queries are
programming instructions and documents are expected assistant replies that may
combine explanation and code.

## Details

### What the Original Data Measures

[CoIR](https://arxiv.org/abs/2407.02883) describes CodeFeedback-ST as a hybrid
code QA retrieval task built from CodeFeedback-Filtered-Instruction. User
instructions are used as queries, and the expected assistant responses are the
documents to retrieve. Unlike `NanoCodeFeedbackMT`, this split tests one-turn
instruction-to-answer matching rather than dialogue-state tracking.

### Observed Data Profile

The Nano split has 200 queries, 10,000 documents, and 200 positive qrels. Each
query has one positive. Queries average 730.51 characters and include Python,
SQL, Linux, graph, and general programming instructions. Documents average
1,538.73 characters and often include runnable snippets plus explanatory text.

### BM25 Difficulty

Using the dataset-provided BM25 candidate column, BM25 reaches nDCG@10 =
0.8755 and hit@10 = 0.9300. It ranks 165 positives first and 186 in the top 10.
The lexical baseline is strong because many instructions repeat the same API,
language, or task words as the answer, though semantic matching is still needed
when answers paraphrase the requested operation.

### Training Data That May Help

Instruction-following code data, StackOverflow-style answers, SQL generation
pairs, and assistant-response retrieval data should help. Hard negatives should
share language and library terms while solving a different requested operation.

### Synthetic Data Guidance

Generate specific programming instructions paired with complete assistant
answers. Include both code-only and prose-plus-code responses, and ensure
negative answers do not merely differ in formatting but solve a different task.
