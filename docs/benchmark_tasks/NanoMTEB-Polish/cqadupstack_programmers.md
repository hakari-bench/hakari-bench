## Overview

`cqadupstack_programmers` is a Polish duplicate-question retrieval task from the
Programmers split of CQADupStack. The queries concern software engineering
practice rather than code implementation alone: testing, architecture, UML,
Scrum, career roles, and project process. Relevant documents are duplicate or
near-duplicate community QA posts.

## Details

### What the Original Data Measures

[CQADupStack: A Benchmark Data Set for Community Question-Answering Research](https://ir.webis.de/anthology/2015.adcs_conference-2015.3/)
defines the source benchmark as duplicate-question retrieval for community QA.
The [MTEB paper](https://arxiv.org/abs/2210.07316) includes CQADupStack as a
retrieval benchmark; the Polish Programmers split is identified through the
MTEB/CLARIN dataset cards.

### Observed Data Profile

The split has 200 queries, 10,000 documents, and 634 positive qrels. Queries
average 59.12 characters and documents average 1,075.30 characters. Examples
include unit testing, solo Scrum, software architecture, UML, and technical
architect roles. Multi-positive clusters are common, with 84 queries having
more than one positive.

### BM25 Difficulty

BM25 reaches nDCG@10 = 0.3241 and hit@10 = 0.4700. It ranks 66 positives first
and 94 in the top 10. Lexical cues such as "UML", "SCRUM", or "test
jednostkowy" are useful, but process and career questions often share broad
software vocabulary while differing in the real duplicate intent.

### Training Data That May Help

Useful data includes non-overlapping software-engineering QA duplicates,
Polish developer forum questions, translated Stack Exchange paraphrases, and
hard negatives about the same role or methodology but a different decision.
Evaluation duplicate clusters and positives should not be used for training.

### Synthetic Data Guidance

Generate Polish software-engineering questions and posts about methodology,
architecture, testing, and project management. Synthetic duplicates should keep
the same practical decision or conceptual issue while varying role, team size,
and example context.
