## Overview

The CQADupStack Mathematica split evaluates whether duplicate community
questions can be recovered across different descriptions of the same Wolfram
Language issue. This Polish version uses translated short queries and longer
candidate posts, many containing code-oriented terminology. The sampled data
mentions rasterized plots, dynamic graphics, common subexpressions, duplicate
list values, and kernel process behavior, so retrieval requires linking
function names, notebook behavior, symbolic manipulation, graphics operations,
and code examples to the same underlying problem.

## Details

### What the Original Data Measures

[CQADupStack: A Benchmark Data Set for Community Question-Answering Research](https://ir.webis.de/anthology/2015.adcs_conference-2015.3/)
is the source benchmark family for duplicate community QA retrieval. The
[MTEB paper](https://arxiv.org/abs/2210.07316) includes CQADupStack in its
retrieval suite; this Polish split is defined by the MTEB/CLARIN dataset cards
and observed Nano data.

### Observed Data Profile

The split has 200 queries, 10,000 documents, and 506 positive qrels. Queries
average 50.41 characters and documents average 1,088.52 characters, making this
one of the longer CQADupStack Polish splits. Examples involve rasterized plots,
dynamic bubble graphics, common subexpressions, duplicate list values, and
kernel process behavior.

### BM25 Difficulty

BM25 reaches nDCG@10 = 0.2129 and hit@10 = 0.3550, with 36 positives at rank 1
and 71 in the top 10. Mathematica identifiers and option names help lexical
retrieval, but many documents contain long code and prose where the duplicate
relation depends on the same computational goal rather than identical words.

### Training Data That May Help

Useful data includes non-overlapping Mathematica Stack Exchange duplicates,
Polish or translated computational notebook QA, function-name paraphrase pairs,
and hard negatives sharing identifiers such as `Plot`, `Dynamic`, or
`DeleteDuplicates` but solving different problems. Exclude evaluation queries,
qrels, and positive posts from any training mixture.

### Synthetic Data Guidance

Create Polish Mathematica posts with realistic code snippets, option names, and
error symptoms, then generate duplicate questions that ask for the same outcome
with different code structure. Synthetic data should preserve executable
semantics and avoid using evaluation positives as seeds.
