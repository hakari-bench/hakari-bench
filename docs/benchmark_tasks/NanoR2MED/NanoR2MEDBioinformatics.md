## Overview

`NanoR2MEDBioinformatics` is an English reasoning-driven medical retrieval task
from R2MED. Queries are Bioinformatics StackExchange posts about genomics,
sequencing, file formats, alignment, variant representation, and analysis tools.
Positive documents are supporting webpages or documentation passages that help
answer the post. The task tests whether a retriever can bridge from a practical
bioinformatics problem to the external technical or biomedical resource that
explains the underlying concept or command behavior.

## Details

### What the Original Data Measures

[R2MED: A Benchmark for Reasoning-Driven Medical Retrieval](https://arxiv.org/abs/2505.14558)
introduces R2MED as a benchmark for medical retrieval where relevance often
depends on a latent reasoning answer rather than surface overlap. The current
arXiv record was first submitted on May 20, 2025 and revised as v2 on April 3,
2026. The paper states that R2MED contains 876 queries across Q&A reference
retrieval, clinical evidence retrieval, and clinical case retrieval.

Bioinformatics belongs to the Q&A reference retrieval group. The paper reports
that the Q&A reference datasets use StackExchange posts as queries and linked
external webpages from accepted or highly upvoted answers as initial positives.
For Bioinformatics, R2MED constructs a new StackExchange-sourced dataset, uses
`wiki_medical_terms` as a domain-relevant negative corpus, chunks documents into
roughly 128-token passages, mines additional candidates with BM25, MedCPT, and
BGE-Large, and then applies GPT-4o relevance assessment plus expert review.

### Observed Data Profile

The Nano split has 77 queries, 10,000 documents, and 226 positive qrel rows.
Queries average 890.32 characters because many posts include commands, sequence
fragments, code blocks, or detailed experimental context. Documents average
666.84 characters and are usually documentation chunks, Wikipedia-style
technical pages, or biomedical article passages.

Each query has 2.94 positives on average, with a median of 2 and a maximum of
8 positives. The sampled questions include Oxford Nanopore alignment quality,
compressed FASTA indexing in STAR, PDB remark fields, VCF-to-BED conversion,
and read counting with bedtools. This is a hybrid scientific and developer
retrieval task: the correct document may be a tool manual, a file-format
definition, or a biology passage.

### BM25 Difficulty

Using the dataset-provided BM25 candidate column, BM25 reaches nDCG@10 = 0.1786
and hit@10 = 0.4416. It ranks 8 positives first and places at least one positive
inside the top 10 for 34 of 77 queries. BM25 is useful when tool names or file
formats are explicit, as in bedtools coverage, but it often misses the reasoning
bridge from a practical question to the specific supporting document.

Observed failures include a SARS-CoV-2 poly(A)-tail question where BM25 retrieves
general SARS pages before the relevant coronavirus polyadenylation passage, a
30x human-genome coverage question where it retrieves read-simulator content
before sequencing-accuracy evidence, and single-cell dropout questions where it
retrieves topic-matching scRNA-seq pages before imputation-method support.

### Training Data That May Help

Useful training data includes non-overlapping Bioinformatics StackExchange
question-to-answer-link pairs, documentation retrieval for sequencing tools,
genomics file-format manuals, biomedical method QA, and hard negatives from
nearby tool pages. BRIGHT-style reasoning-intensive retrieval data may also help
because R2MED explicitly builds on the same idea of retrieving answer-supporting
references rather than paraphrases.

Training should not use the R2MED Bioinformatics evaluation queries, qrels, or
positive passages. When using StackExchange dumps or tool documentation, remove
obvious overlap with the evaluated posts and linked pages.

### Synthetic Data Guidance

For document-to-question generation, start from non-evaluation tool manuals,
file-format specifications, or biomedical methods passages and generate realistic
developer-scientist questions that require using that passage to solve an
analysis problem. Include command snippets, error symptoms, and biological
context when they are grounded in the document.

For joint generation, create short documentation chunks plus StackExchange-style
questions about why an alignment, indexing, variant conversion, or sequencing
metric behaves a certain way. Hard negatives should share tool names or data
types but answer a different operation. Do not seed synthetic data with Nano
evaluation queries or positives.
